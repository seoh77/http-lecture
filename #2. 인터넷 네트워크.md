# #2. 인터넷 네트워크

### 인터넷 통신

- 클라이언트 ↔ 인터넷 ↔ 서버
- 인터넷 망을 통해서 메시지를 전달

<br />

## IP (인터넷 프로토콜)

#### IP (인터넷 프로토콜) 역할

- 지정한 IP 주소 (IP Address)에 데이터 전달
- 패킷(Packet)이라는 통신 단위로 데이터 전달

#### 패킷 전달

- IP 패킷 정보
  - 출발지 IP, 목적지 IP, 데이터 등을 담음
- 송신자의 출발지 IP와 수신자의 목적지 IP를 넣어 주고 인터넷 망에 넘겨주면 노드를 거쳐서 서버까지 전달
- 잘 받았으면 수신자가 송신자에게 잘 받았다고 메시지 전달

#### IP 프로토콜의 한계

- 비연결성
  - 패킷을 받을 대상이 없거나 서비스 불능 상태여도 패킷 전송
  - 대상 서버가 패킷을 받을 수 있는 상태인지 모름
- 비신뢰성
  - 중간에 패킷이 손실되는 문제가 발생할 수 있다.
  - 순차적 패킷 전송이 보장X  
    ex. 패킷이 큰 경우 보통 1500바이트씩 끊어서 전송한다. 이때 a를 보내고 b를 보내는 경우 이동하는 노드의 경로가 다를 수 있다. 의도는 a, b 순으로 도착하는 거지만 b, a 순으로 도착할 수 있다.
- 프로그램 구분
  - 같은 IP를 사용하는 서버에서 통신하는 애플리케이션이 여러 개일 수도 있다.
  - 예를 들어 게임을 하면서 음악을 듣는 등 같은 IP에서 여러개의 애플리케이션 동작시 구분되지 않는다.

## TCP와 UDP

#### 인터넷 프로토콜 스택의 4계층

- 애플리케이션 계층 : HTTP, FTP  
  → 프로그램이 'Hello, world!' 메시지를 생성하고, 소켓 라이브러리를 통해 전달
- 전송 계층 : TCP, UDP  
  → TCP 정보 생성, 메시지 데이터 포함
- 인터넷 계층 : IP  
  → IP 패킷 생성, TCP 데이터 포함
- 네트워크 인터페이스 계층(랜카드, 랜드라이버 등)

<br />

### TCP

#### TCP/IP패킷 정보

- 출발지IP, 목적지IP, 데이터 등
- 출발지 PORT, 목적지 PORT
- 전송 제어, 순서, 검증 정보 등의 내용이 포함  
  → IP만으로 해결되지 않았던 문제들이 해결

#### TCP 특징

- TCP (Transmission Control Protocol) : 전송 제어 프로토콜
- 연결 지향
  - TCP 3 way handshake (가상 연결)
  - 먼저 연결한 후 메시지를 보냄
- 데이터 전달 보증
  - 패킷을 보냈을 때 중간에 누락되었는지 여부를 알 수 있음
- 순서 보장

→ 신뢰할 수 있는 프로토콜로, 현재는 대부분 TCP를 사용

#### TCP 3 way handshake

1. 클라이언트에서 서버로 SYN 메시지 전달
2. 서버가 클라이언트에게 SYN + ACK 메시지 전달
3. 클라이언트가 서버에게 ACK 메시지 전달
4. 데이터 전송  
   → 요즘은 최적화가 돼서 3번 ACK를 보낼 때 데이터도 함께 전송한다.

- 물리적으로 연결된 것이 아니라 개념적으로 연결되는 것이다. (나를 위한 전용 랜선이 보장되는 것이 아니다)
- 위 방법으로 연결하면 클라이언트가 서버를 믿을 수 있으며, 서버가 클라이언트를 믿을 수 있다.
- 연결이 맺어진 다음에 데이터 전달

#### 순서 보장

- 패킷 1번 다음 2번이 아닌 다른 패킷이 오면 다 버리고 2번부터 다시 보내도록 요청
- 내부적으로 최적화가 일어날 수는 있지만 기본적으로 이런 원리

→ TCP/IP 패킷 정보에 전송 제어, 순서 등의 데이터가 포함되어 있기 때문에 가능

### UDP

#### UDP 특징

- UDP (User Datagram Protocol) : 사용자 데이터그램 프로토콜
- 기능이 거의 없다.
- 연결지향 X (TCP 3 way handshake X)
- 데이터 잔달 보증 X
- 순서 보장 X
- 데이터 전달 및 순서가 보장되지 않지만, 단순하고 빠름
- IP와 거의 같지만, PORT와 체크섬 정도만 추가
  - 하나의 IP에 여러개의 패킷이 올 때 PORT를 이용하여 작업 구분 가능
  - 게임과 음악을 동시에 사용하더라도 서로 다른 PORT를 사용하기 때문에 구분할 수 있다.
  - 체크섬은 메시지가 제대로 왔는지 검증해주는 데이터
- 애플리케이션에서의 추가 작업 필요

#### UDP 장점

- TCP에 비해 전송 속도가 빠르며 최적화할 수 있다.
  - TCP 프로토콜은 손을 댈 수 없음
- TCP는 그래로 사용하고, UDP는 내가 원하는대로 애플리케이션 레벨에서 만들어준다.
- HTTP3 같은 경우 UDP를 사용하며 최근에는 UDP가 뜨고 있다.

## PORT

- 한 번에 둘 이상을 연결해야 한다면?
  - 게임, 화상통화, 웹 브라우저 요청 등
  - 하나의 클라이언트가 여러개의 서버와 통신해야 한다.
- IP에 더해서 PORT
  - IP는 목적지 (아파트)
  - PORT는 각각의 애플리케이션 구분 (각각의 호)
  - 서버가 내 PORT를 알 수 있는 이유는 패킷에 출발지 PORT도 써준다.
- 0 ~ 65535 할당 가능
  - 0 ~ 1023 : 잘 알려진 포트, 사용하지 않는 것이 좋음
  - FTP - 20, 21
  - TELNET - 23
  - HTTP - 80
  - HTTPS - 443

## DNS

- IP는 기억하기 어렵고, 변경될 수 있다는 문제점이 있다.
- DNS (Domain Name System) : 도메인 네임 시스템
  - 전화번호부
  - 도메인 명을 IP주소로 변환
- DNS 서버에 도메인 명과 IP주소를 등록해두면 DNS 서버가 도메인 이름에 해당하는 IP주소를 반환
  - IP 주소를 바꾸더라도 도메인 명은 그대로다.
